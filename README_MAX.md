# Бот учёта времени (MAX)

Отдельная точка входа и отдельная SQLite-база рядом с Telegram-ботом в этом репозитории. Telegram (`main.py`) не меняется.

## Запуск

```bash
pip install -r requirements.txt
python main_max.py
```

Нужен файл `.env` в корне проекта (см. ниже).

### Linux (systemd)

В репозитории есть юнит [`work_max.service`](work_max.service) — по аналогии с `work.service` для Telegram. Пути по умолчанию: `/root/WorkTimeChecker` и venv `venv`. Под себя отредактируйте `WorkingDirectory` и `ExecStart`, затем:

```bash
sudo cp work_max.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now work_max.service
sudo systemctl status work_max.service
```

Логи: `journalctl -u work_max.service -f`.

## Логи

По умолчанию логи **только в консоль** (терминал или stdout/stderr systemd), папка `logs/` может быть пустой — это нормально.

| Бот        | Куда смотреть |
| ---------- | ------------- |
| **MAX**    | Консоль; под systemd — `journalctl -u work_max.service -f`. Файл: включите `LOG_TO_FILE=true` в `.env` → `logs/max_bot.log`. |
| **Telegram** | Консоль; файл: `LOG_TO_FILE=true` → `logs/bot.log` (structlog). |

## Переменные окружения (`.env`)


| Переменная                    | Обязательно | Описание                                                                                                         |
| ----------------------------- | ----------- | ---------------------------------------------------------------------------------------------------------------- |
| `MAX_BOT_TOKEN`               | да          | Токен бота MAX                                                                                                   |
| `MAX_API_BASE_URL`            | нет         | Базовый URL REST API (по умолчанию `https://platform-api.max.ru`)                                                |
| `MAX_DB_PATH`                 | нет         | Файл SQLite; по умолчанию `base_max.sqlite` в корне проекта                                                      |
| `MAX_GROUP_CHAT_ID`           | нет         | Только **chat_id** групп/каналов (`/id` в этом чате). **Не** подставляйте `user_id` сотрудника — будет `chat.not.found`. Несколько id через запятую. |
| `MAX_BROADCAST_USER_IDS`      | нет         | **user_id** людей в MAX: копии той же сводки (конец смены, отпуска) уходят в личку через `?user_id=`. Несколько через запятую. |
| `MAX_ADMIN_IDS`               | нет         | Через запятую **user_id людей** в MAX (не id бота). Стартовое уведомление «бот запущен»                          |
| `MAX_REPORT_INTERVAL_MINUTES` | нет         | Тест: если задано число (например `5`), все рассылки подряд раз в N минут вместо cron. Пусто — боевое расписание |
| `LOG_TO_FILE`                 | нет         | `true` — дополнительно писать логи MAX в `logs/max_bot.log` (в консоль вывод не отключается)                      |


Узнать `user_id` и `chat_id` группы: команда `**/id`** в нужном чате.

## Команды в MAX


| Команда  | Назначение                                                             |
| -------- | ---------------------------------------------------------------------- |
| `/start` | Сценарий учёта времени (как в Telegram-версии): смена, перерыв, отпуск |
| `/id`    | Показать ваш `user_id` и `chat_id` текущего чата                       |


Учёт ведётся в личке с ботом (не в группе), если тип чата передаётся в событии.

## Расписание рассылок

**Боевой режим** (без `MAX_REPORT_INTERVAL_MINUTES`):

- Утро: 7:59, 8:15, 8:30, 8:45, 9:00 — напоминание выйти на работу  
- 17:00 — вечерняя рассылка  
- 18:00 — отпуска: в чаты (`MAX_GROUP_CHAT_ID`) и/или в ЛС (`MAX_BROADCAST_USER_IDS`)  
- 23:55 — завершение дня / автозакрытие смен (как у Telegram-бота)

**Тестовый режим**: `MAX_REPORT_INTERVAL_MINUTES=5` — подряд вызываются утро, вечер, end_task и отпуск, каждые 5 минут. Первый такой тик выполняется сразу после старта. Админам из `MAX_ADMIN_IDS` в конце каждого тика уходит **текстовая сводка** (сколько человек в каждой категории), чтобы было видно, что планировщик живой, даже если личных напоминаний ноль.

## Что именно уходит в «отчётах» (рассылках)

Это **не произвольный отчёт по закрытой смене**, а **напоминания по состоянию дня**:


| Этап         | Кому уходит                            | Условие                                                                                          |
| ------------ | -------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **Утро**     | Сотруднику в ЛС                        | В БД: `is_worked = 1`, не в отпуске, и **сегодня ещё нет времени начала смены** (`begin` пусто). |
| **Вечер**    | Сотруднику в ЛС                        | **Сегодня есть `begin`, но нет `end`** — смена считается открытой.                               |
| **end_task** | Дозакрытие + сводка                    | Итог смены: в каждый **chat_id** из `MAX_GROUP_CHAT_ID` и каждому **user_id** из `MAX_BROADCAST_USER_IDS`. |
| **Отпуск**   | Чаты и/или ЛС                          | По каждому отпускнику — то же сообщение во все настроенные получатели. |


Если вы **уже начали и закончили смену за сегодня**, вы **не входите** ни в утренний, ни в вечерний список — **личных напоминаний не будет**. Это ожидаемо. Итог смены при «закончить смену» уходит в **группы** (`MAX_GROUP_CHAT_ID` = только `chat_id` чата) и **в личку** тем, кто перечислен в `MAX_BROADCAST_USER_IDS` (`user_id`). Это разные поля API: `user_id` из колбэка не равен `chat_id` лички (см. вывод `/id`).

Чтобы снова попасть в утреннюю рассылку, нужен **новый календарный день** (или сброс записи смены за день в БД).

## База данных и сотрудники

- Файл по умолчанию: `base_max.sqlite`  
- Таблицы: `users_max`, `works_max`  
- Предзаполнение ФИО: `workers_max.json` — формат как у `workers.json`, в `user_id` указывается **числовой id пользователя в MAX**; пустой `user_id` в строке игнорируется при сиде

## HTTP API (FastAPI)

Сервис поднимается из `api.py` (тот же процесс для Telegram и MAX):

```bash
python api.py
```

По умолчанию в коде указан хост `0.0.0.0`, порт **8000** (при необходимости измените в `api.py`).

### Эндпоинты, относящиеся к MAX

Все методы — **GET**, ответ **JSON**.

#### `GET /getdata_max`

Смены из `base_max.sqlite` за период.

**Query-параметры:**


| Параметр    | Формат       | Описание       |
| ----------- | ------------ | -------------- |
| `from_date` | `YYYY-MM-DD` | Начало периода |
| `to_date`   | `YYYY-MM-DD` | Конец периода  |


**Ответ:** массив объектов (схема как у Telegram `/getdata`):


| Поле          | Тип           | Описание                                             |
| ------------- | ------------- | ---------------------------------------------------- |
| `date`        | string        | Дата смены, `дд.мм.гггг`                             |
| `start`       | string        | Начало, `дд.мм.гггг ЧЧ:ММ`                           |
| `end`         | string | null | Конец или `null`                                     |
| `dinners_min` | string        | Перерывы, минуты (целая часть от секунд в БД)        |
| `usernameTG`  | string | null | Для MAX — **username в MAX** (имя поля историческое) |
| `user_id`     | int           | Внутренний id в `users_max`                          |


**Пример:**

```http
GET http://localhost:8000/getdata_max?from_date=2026-04-01&to_date=2026-04-30
```

#### `GET /users_max`

Список пользователей из `users_max`.

**Ответ:** массив объектов:


| Поле            | Тип           | Описание                       |
| --------------- | ------------- | ------------------------------ |
| `id`            | int           | Первичный ключ                 |
| `max_user_id`   | string        | User id в MAX                  |
| `username`      | string | null | Username                       |
| `name`          | string | null | Имя из профиля                 |
| `fio`           | string | null | ФИО                            |
| `is_active`     | int           | Флаг                           |
| `is_worked`     | int           | Участвует в утренних рассылках |
| `vacation_to`   | string | null | Конец отпуска `дд.мм.гггг`     |
| `register_date` | string | null | Регистрация `дд.мм.гггг ЧЧ:ММ` |


**Пример:**

```http
GET http://localhost:8000/users_max
```

Примечание: при первом вызове эндпоинтов MAX подгружается модуль `database.db_max` — в `.env` должен быть валидный `MAX_BOT_TOKEN` (как при запуске бота).

## Внешний API мессенджера MAX

Исходящие запросы бота (библиотека **maxapi** и модуль `max_app/messaging.py`) идут на:

`{MAX_API_BASE_URL}/messages`  

методы: отправка сообщений (в т.ч. с inline-кнопками), удаление по `message_id`.
